# Design Log #137 â€” Production Build Self-Containment

## Background

When deploying to production (Docker/Cloud Run), only `build/`, `config/`, and `node_modules/` are in the container â€” no source files (`src/`). The production server fails in multiple places because it tries to read source files that don't exist. The build output should be fully self-contained.

### Related Design Logs

- #111 â€” Nested headfull full-stack components (template injection, headless pipeline)
- #123 â€” Deeply nested headfull and headless components (recursive hoisting)
- #94 â€” SSR streaming renderer (server element compilation)

## Problem

The production server requires source files that don't exist in the container. Four gaps were discovered during Cloud Run deployment of the golf project:

### Gap 1: Headfull component source files (ENOENT)

Two serve-time operations try to read headfull component source files:

1. **`parseJayFile`** (load-production-parts.ts:61) encounters `<script type="application/jay-headfull" src="../../components/header">` and calls `importResolver.readJayHtml()` â€” fails with ENOENT
2. **`injectHeadfullFSTemplates`** (load-production-parts.ts:154) also reads the same source file â€” fails with ENOENT

Neither operation should require source files. The headfull templates are already inlined inside `<jay:Name>` tags, slow-phase data is already baked in, and the `<script type="application/jay-headfull">` tags are leftover metadata.

Subtlety: **headfull components can declare their own headless imports** (e.g., a header component using a cart-indicator plugin). `parseHeadfullFSImports` hoists these into the page-level `headlessImports` array at parse time (DL#123), but this hoisting happens only in memory â€” the nested headless imports are NOT written as `<script type="application/jay-headless">` tags in the pre-rendered HTML.

### Gap 2: CSS files referenced by headfull components (ENOENT)

Headfull component jay-html files reference CSS via `<link rel="stylesheet" href="../../styles/header.css">`. Since `parseJayFile` processes these at serve time, it resolves the relative path to `src/styles/` â€” which doesn't exist.

The CSS is already compiled into the build output (per-route bundles in `build/v1/pre-rendered/`). The source CSS reference in the pre-rendered jay-html is stale.

**Fix:** Strip `<link>` tags that reference source CSS from the pre-rendered jay-html during build. The compiled CSS is already emitted by the build pipeline.

### Gap 3: Materialized contracts needed at serve time (ENOENT)

Dynamic contracts (e.g., `product-page`) are generated by `jay-stack agent-kit` and stored in `agent-kit/materialized-contracts/`. The production server's `parseJayFile` resolves contracts at serve time and looks for them there.

**Fix:** The build pipeline should embed the contract data needed at serve time into the pre-rendered output or the route manifest â€” not require the `agent-kit/` directory at runtime. Alternatively, copy materialized contracts into the build output directory during build.

### Gap 4: Absolute paths in route manifest

`route-manifest.json` bakes in the build machine's absolute path as `projectRoot` (e.g., `/Users/yoav/work/jay/golf`). The production server passes this to `scanPlugins()` for plugin discovery. Inside the container, that path doesn't exist.

**Fix:** The production server should use `process.cwd()` (or a CLI flag) instead of `manifest.projectRoot` for plugin discovery. Or the manifest should use relative paths.

## Questions and Answers

### Q1: After slow rendering, should the headfull component remain as a headfull import in the pre-rendered jay-html?

No. At that point, the headfull component has contributed everything it can:

| Contribution        | Where it lives after slow rendering                  |
| ------------------- | ---------------------------------------------------- |
| Template content    | Inlined in `<jay:Name>` tags in the body             |
| Slow-phase bindings | Resolved into static HTML                            |
| Component module    | Referenced by compiled server-element/hydrate output |

The `<script type="application/jay-headfull">` tag serves no purpose in the pre-rendered output.

### Q2: Should headfull imports be converted to headless imports?

No. A headfull component is not a headless component â€” converting the import type would be semantically wrong. What we need is to **hoist the headfull component's nested headless imports** up to the page level, then remove the headfull tag entirely.

### Q3: What about the headfull component's own headless imports?

This is the critical detail. A headfull component like `kitan-header` may declare:

```html
<!-- components/kitan-header.jay-html -->
<head>
  <script type="application/jay-headless" src="@wix/cart-indicator-plugin">
    cart-indicator: cartIndicator
  </script>
</head>
```

During build, `parseHeadfullFSImports` hoists this into the page's `headlessImports` array. But the hoisting only happens in memory â€” it's not persisted in the pre-rendered HTML. If we strip the headfull tag without persisting these, the production server won't discover `cart-indicator` as a headless import.

**Fix:** During the build pipeline, after slow rendering, write the headfull component's headless imports as page-level `<script type="application/jay-headless">` tags, then strip the `<script type="application/jay-headfull">` tag.

## Design

### What changes in the build pipeline

In `instance-pipeline.ts`, after slow rendering produces `preRenderedJayHtml` (line 167) and before writing to disk (line 238):

1. Parse the pre-rendered HTML
2. For each `<script type="application/jay-headfull">` tag:
   a. Read the headfull component's jay-html (still accessible at build time)
   b. Extract any `<script type="application/jay-headless">` tags from its `<head>`
   c. Append those as new `<script type="application/jay-headless">` tags in the page's `<head>` (skip duplicates already present)
   d. Remove the `<script type="application/jay-headfull">` tag
3. Write the cleaned pre-rendered HTML

Deduplication: if the page already imports the same headless component (same `src` and key), skip it. Compare by the `src` attribute value.

### What changes in load-production-parts.ts

With headfull tags stripped from pre-rendered HTML:

- `parseJayFile` (line 61) â€” no headfull imports to resolve, works without source files
- `injectHeadfullFSTemplates` (line 154) â€” finds no headfull elements, returns input unchanged (could also be removed entirely for clarity)

The headless imports that were nested inside headfull components are now page-level headless tags, so `parseJayFile` discovers them normally.

### Recursive headfull nesting (DL#123 Scenario B)

If headfull component A includes headfull component B, `parseHeadfullFSImports` already handles this recursively â€” all nested headless imports are hoisted to the page level. The build pipeline should process recursively too: when reading headfull A's jay-html to extract headless imports, it should also process headfull B's nested headfull imports. Since `injectHeadfullFSTemplates` already handles recursive injection at build time, the templates are already inlined â€” we just need the headless import hoisting.

Simplest approach: call `parseHeadfullFSImports` on the pre-rendered jay-html at build time, collect all headless imports it discovers from headfull components, add them as tags, then strip headfull tags. This reuses existing recursive logic.

```
Build time (instance-pipeline.ts):
  1. injectHeadfullFSTemplates() â†’ templates inlined     [already done]
  2. slowRenderTransform() â†’ slow bindings resolved       [already done]
  3. hoistHeadfullHeadlessImports() â†’ add headless tags   [NEW]
  4. stripHeadfullTags() â†’ remove headfull script tags    [NEW]
  5. writeFile() â†’ write pre-rendered jay-html            [already done]
```

## Implementation Plan

### Phase 1: Hoist headfull headless imports and strip headfull tags (Gap 1)

1. In `instance-pipeline.ts`, after slow rendering (line 167), add a new step before writing to disk
2. Parse the pre-rendered HTML with node-html-parser
3. For each `<script type="application/jay-headfull">` tag:
   - Read the headfull component's jay-html via `JAY_IMPORT_RESOLVER.readJayHtml()`
   - Extract any `<script type="application/jay-headless">` tags from its `<head>`
   - Append those as page-level `<script type="application/jay-headless">` tags (skip duplicates by `src`)
   - Handle recursive headfull nesting
   - Remove the `<script type="application/jay-headfull">` tag
4. Serialize the modified HTML back to string

### Phase 2: Strip source CSS references (Gap 2)

1. In the same build pipeline step, remove `<link rel="stylesheet">` tags whose `href` resolves to a source file path (outside `build/`)
2. The compiled CSS is already emitted per-route by the build pipeline â€” no information is lost

### Phase 3: Embed materialized contracts in build output (Gap 3)

1. During `jay-stack build`, copy materialized contracts into `build/v1/contracts/` (or embed contract data in the route manifest per-route)
2. Update `loadProductionPageParts` to resolve contracts from the build directory instead of `agent-kit/materialized-contracts/`

### Phase 4: Relative paths in route manifest (Gap 4)

1. Write relative paths (relative to manifest location) in `route-manifest.json` instead of absolute
2. Update `startMainServer` to resolve relative paths against `process.cwd()` or the manifest's directory
3. Remove `projectRoot` from the manifest (or make it optional, defaulting to `cwd`)

### Phase 5: Clean up load-production-parts.ts

1. Remove the `injectHeadfullFSTemplates` call (line 154) â€” no longer needed since headfull tags are gone and templates are already inlined
2. Optionally add a guard: if `parseJayFile` still encounters a headfull import (e.g., stale cache), log a warning instead of crashing

## Trade-offs

| Decision                               | Pro                                              | Con                                                             |
| -------------------------------------- | ------------------------------------------------ | --------------------------------------------------------------- |
| Hoist at build time, not serve time    | Production server needs zero source files        | Build pipeline becomes slightly more complex                    |
| Hoist as `<script>` tags, not metadata | `parseJayFile` works unchanged                   | Slightly redundant with in-memory hoisting                      |
| Strip headfull tags entirely           | Clean separation: build-time vs serve-time       | Pre-rendered HTML no longer shows headfull component provenance |
| Relative paths in manifest             | No `sed` workaround in Dockerfile                | Manifest consumers must resolve relative paths                  |
| Embed contracts in build output        | Self-contained build, no `agent-kit/` at runtime | Contracts duplicated between `agent-kit/` and `build/`          |
